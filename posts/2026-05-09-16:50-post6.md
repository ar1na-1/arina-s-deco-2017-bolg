# Evaluation, Reflection, and Future Development

## Project Goal

The goal of this project was to design a community platform that improves the quality of peer feedback.

Instead of allowing feedback to remain broad or disconnected from the poster’s intention, the prototype encourages users to define a clear feedback goal and respond through structured prompts.

The final Assignment 2 prototype supports this goal through a connected multi-page flow.

## Prototype Evaluation

The final prototype supports two main user journeys.

### Journey 1: Reviewing Community Posts

Users can browse the home feed, open a post detail page, view the prototype image, read the project context, and provide structured feedback.

This journey supports reviewers by giving them enough context before they respond.

The three feedback prompts help users focus on:

- What feels clear
- What feels confusing
- What could be improved

This makes the feedback more specific and easier for the original poster to interpret.

### Journey 2: Creating a New Post

Users can create a new post by entering a title, feedback goal, project description, and prototype image.

They then move to the Feedback Goal Checklist before publishing.

This extra step encourages users to reflect on whether their request is clear and specific.

## What Worked Well

The most successful part of the prototype is the connection between the feedback goal and the feedback form.

The home feed introduces each post through its title, image, category, and goal. The detail page then provides more context before asking users to respond.

This creates a clearer relationship between what the poster needs and what the reviewer provides.

The multi-page structure also worked well. Separating the home feed, detail page, create post page, and checklist page helped reduce visual overload and made each task easier to understand.

## Limitations

The prototype still has several limitations.

First, the feedback submission is simulated rather than stored dynamically. This means users can experience the interaction flow, but the system does not yet save new feedback into a database.

Second, the create post flow does not yet add a new post to the home feed. The current prototype demonstrates the intended interaction, but does not fully implement persistent content creation.

Third, the platform does not yet include advanced discovery tools such as filtering, sorting, or saved posts. These would become more important if the community had many posts.

## Accessibility Considerations

Accessibility was considered through the use of clear page structure, visible buttons, labelled form fields, and consistent spacing.

The interface uses large clickable areas and clear visual hierarchy to make the main tasks easier to understand.

However, future improvements should include:

- Stronger keyboard navigation testing
- More detailed alternative text for images
- Clear focus states for buttons and links
- Further colour contrast testing

These improvements would make the prototype more inclusive and reliable for a wider range of users.

## Future Development

If the project were developed further, the next step would be implementing persistent data storage.

This would allow users to:

- Submit real feedback
- Add new posts to the feed
- View feedback history
- Track their own posts

Another future improvement would be adding filtering or search by feedback goal. This would help users find related projects and learn from similar feedback requests.

User profiles could also support stronger community identity, allowing users to see their posts, feedback activity, and saved discussions.

## Reflection

This project helped me understand that designing a community platform is not only about creating pages, but about designing the relationship between users, content, and interaction.

The most important design decision was requiring users to define a feedback goal before receiving responses.

This small structural change shaped the entire prototype, from the home feed to the detail page and create post flow.

The project also helped me understand how wireframes, content structure, and front-end implementation need to align. If the prototype structure does not match the original design goal, the interaction becomes unclear.

## Conclusion

The final prototype successfully demonstrates the concept of a goal-oriented feedback community.

Although it does not yet include persistent data or full backend functionality, it communicates the main interaction model clearly.

By combining goal-based posting, structured feedback prompts, and a checklist before publishing, the platform supports more focused, contextual, and actionable community feedback.
