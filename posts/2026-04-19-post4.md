# Designing the System Structure and Data Model

## Problem Context

After defining the interaction flow, the next challenge was designing a system structure that could support the actual behaviours implemented in Assignment 2.

The prototype allows users to browse community posts, view post details, submit structured feedback, and create new feedback requests.

To support these actions consistently, the system needed a simple but clear data structure.

## Problem Analysis

A major challenge was organising relationships between users, posts, and feedback.

Without a clear structure, feedback could become disconnected from its original post, making it difficult for users to understand the context of comments.

Another issue was scalability.

Even though the current prototype uses static example content, the system should still reflect how real community data would be stored and connected.

## Design Decision

The final system was organised around three core entities.

### Entity 1: User

Each user represents a community participant.

A user can:

- Create posts  
- Review other posts  
- Submit feedback  

Key attributes:

- user_id  
- username  

### Entity 2: Post

A post represents a design idea or project shared by a user.

Each post contains:

- post_id  
- author_id  
- title  
- description  
- feedback_goal  
- category  
- image  

This matches the current prototype, where every post displays project information before feedback is requested.

### Entity 3: Feedback

Feedback is connected to both a user and a post.

Each feedback entry contains:

- feedback_id  
- post_id  
- reviewer_id  
- clarity  
- confusion  
- suggestion  

This supports the structured feedback model implemented in the prototype.

## System Relationships

The relationships are:

**One User → Many Posts**

**One Post → Many Feedback Entries**

**One User → Many Feedback Entries**

This allows the platform to preserve context while supporting community interaction.

## Alternative Approaches

One alternative would be storing feedback as a single text comment.

This would reduce implementation complexity, but it would make feedback harder to compare and analyse.

Another alternative would be introducing a more complex scoring system.

While this may support deeper evaluation, it would increase both system complexity and user effort.

## Trade-offs

The chosen data model improves clarity, consistency, and future scalability.

However, it also reduces flexibility, since users must provide feedback using predefined categories rather than open comments.

Despite this limitation, the structure better supports the project goal of meaningful, goal-oriented feedback.

## Conclusion

Designing the system structure was essential for turning interaction concepts into a functional prototype.

By connecting users, posts, and feedback through a simple relational model, the platform supports clear, contextual, and actionable community interaction.
